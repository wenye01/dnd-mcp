## 1. 背景与目标

### 1.1 背景

DND MCP 项目采用 Client-Server 架构：

**Client 现状**：
- 已完成基础框架：会话管理、对话历史、WebSocket 推送
- 使用 Redis + PostgreSQL 存储
- 提供 HTTP API 供前端调用
- **已有 MCP Client 接口**（`internal/mcp`），用于连接 Server
- Session 模型包含 `MCPServerURL` 字段，预留了 Server 连接

**Server 待开发**：
- 可复用的 D&D 5e 游戏规则引擎
- 通过 MCP 协议暴露游戏能力（Tools）
- 独立服务，可被任意 MCP Client 使用

### 1.2 Client 变更影响

由于 Server 统一管理游戏数据和对话历史，Client 需要相应调整：

| 变更点 | Client 现状 | Server 设计 | 影响 |
|--------|-------------|-------------|------|
| 对话存储 | Client 存储消息 | Server 统一存储 | Client 变成协调层 |
| 游戏数据 | 无 | Server 管理 | 新增调用 Server Tools |
| LLM 编排 | Client 调用 LLM | 保持不变 | 无影响 |
| WebSocket | Client 推送 | 保持不变 | 无影响 |

**Client 需要的调整**（不在本次设计范围）：
1. 移除消息存储逻辑，改为调用 Server 的 `save_message`
2. 新增调用 Server 的上下文获取接口
3. 新增调用 Server 的游戏规则 Tools

### 1.3 目标

**主要目标**：
- 设计一个独立、可复用的 D&D 5e 游戏规则引擎
- 通过 MCP 协议暴露游戏能力（Tools）
- 与 Client 形成清晰的职责边界
- Server 统一管理对话历史和游戏数据

**成功标准**：
- Server 不依赖 LLM，保持轻量
- 规则引擎可独立测试
- 支持 Client 的两种模式（完整模式/简化模式）
- 战役数据可持久化和恢复
- 可被其他 MCP Client（如 Claude Desktop）复用

### 1.4 范围

**包含**：
- 规则引擎层（角色、战斗、骰子、地图）
- 上下文管理层（战役、对话存储、MCP 适配）
- 战斗系统（回合制、先攻、攻击、施法）
- 地图系统（大地图、战斗地图、移动）
- 骰子与检定系统
- 对话历史存储与上下文压缩

**不包含**：
- 规则可选/冲突处理（内置一套规则）
- 地图编辑功能
- 战斗 AI（由 Client LLM 决定）
- 静态数据管理（法术、物品等，依赖 RAG）
- 状态效果系统（后续扩展）

## 2. 架构与模块

### 2.1 技术选型

| 决策项 | 选择 | 理由 |
|--------|------|------|
| **语言** | Go 1.24+ | 与 Client 技术栈一致 |
| **协议** | MCP 协议 + HTTP 传输 | 标准 LLM 工具协议 |
| **存储** | PostgreSQL | 关系型数据，复杂查询 |
| **架构模式** | Handler → Service → Store | 与 Client 保持一致 |
| **LLM 依赖** | 无 | 保持简单，LLM 能力由 Client 提供 |

### 2.4 模块划分

| 模块 | 职责 | 边界说明 |
|------|------|----------|
| **MCP Adapter** | MCP 协议适配，Tool 定义与注册 | 处理协议细节，不包含业务逻辑 |
| **Campaign** | 战役生命周期管理 | 战役元数据，不包含角色/战斗细节 |
| **Character** | 角色数据管理（含 NPC） | 属性、HP、装备，不包含战斗逻辑 |
| **Combat** | 战斗状态和流程管理 | 回合、先攻、参战者，依赖 Character |
| **Dice** | 骰子投掷和检定计算 | 纯计算，无状态，可独立测试 |
| **Map** | 大地图和战斗地图管理 | 地图数据、位置、移动，不包含战斗 |
| **Context** | 对话历史和上下文管理 | 存储消息，提供滑动窗口压缩 |
| **Lookup** | 规则查询（法术、物品等） | 依赖外部 RAG，只读查询 |

### 2.5 模块依赖关系

```
┌─────────────────────────────────────────────────────────────────────┐
│                         模块依赖关系                                 │
└─────────────────────────────────────────────────────────────────────┘

                              MCP Adapter
                                   │
           ┌───────────────────────┼───────────────────────┐
           │                       │                       │
           ▼                       ▼                       ▼
      Campaign              Character                Context
           │                       │                       │
           │         ┌─────────────┼─────────────┐         │
           │         │             │             │         │
           │         ▼             ▼             ▼         │
           │      Combat    ───►  Dice   ◄──►  Map        │
           │         │             │                       │
           │         │             │                       │
           └─────────┴─────────────┴───────────────────────┘
                                   │
                                   ▼
                               Lookup
                             (外部 RAG)

依赖说明:
  • Combat → Character: 战斗需要角色数据（属性、HP、AC）
  • Combat → Dice: 战斗涉及攻击检定、伤害骰
  • Combat → Map: 战斗需要位置信息（距离、掩护）
  • Map → Dice: 移动可能涉及检定（隐匿等）
  • Context: 独立模块，为 Client 提供对话存储和上下文压缩
  • Lookup: 独立模块，调用外部 RAG 服务查询规则

## 4. 核心概念

### 4.1 术语定义

| 术语 | 定义 | 示例 |
|------|------|------|
| **战役 (Campaign)** | 一次完整的 D&D 游戏会话，包含角色、地图、战斗等所有数据 | "龙之追寻"战役 |
| **角色 (Character)** | 游戏中的实体，包括玩家角色和 NPC | Legolas（精灵游侠） |
| **NPC** | 非玩家角色，通过 `is_npc` 字段区分 | 商人、敌人、村民 |
| **战斗 (Combat)** | 回合制的战斗遭遇，管理先攻顺序和回合 | 旅店遭遇哥布林 |
| **先攻 (Initiative)** | 决定战斗中行动顺序的数值 | Legolas 先攻 18 |
| **大地图 (World Map)** | 世界/区域级别的地图，展示地点和队伍位置 | 中土大陆地图 |
| **战斗地图 (Battle Map)** | 详细场景地图，格子化，支持战斗 | 跃马旅店大厅 |
| **Token** | 战斗地图上的位置标记，代表角色或物体 | Legolas 位于 (3, 5) |
| **上下文 (Context)** | LLM 对话所需的背景信息，包含对话历史和游戏状态 | 最近 20 条消息 + 状态摘要 |
| **滑动窗口** | 上下文压缩策略，保留最近 N 条消息 | 默认 20 条 |
| **格子 (Grid)** | 地图的基本单位，用于位置计算 | 5 英尺 × 5 英尺 |

### 4.2 关键决策

| 决策 | 选择 | 理由 | 风险 |
|------|------|------|------|
| **Server LLM 依赖** | 不依赖 | 保持简单，LLM 由 Client 提供 | 无 |
| **对话存储位置** | Server 统一存储 | 多 Client 数据一致性 | 存储压力 |
| **上下文压缩** | 滑动窗口 | 简单有效，无需 LLM | 可能丢失重要信息 |
| **地图格式** | 统一格子系统 | 复用逻辑，简化实现 | 需要自定义格子内容 |
| **NPC 处理** | 与角色共享结构 | 代码复用，统一管理 | 需要区分逻辑 |
| **战斗 AI** | 不实现 | 保持 Server 简单，由 Client LLM 决定 | 无 |
| **静态数据** | 外部 RAG | 避免 Server 膨胀 | 依赖外部服务 |

## 5. MCP Tools 总览

### 5.1 战役管理

| Tool | 描述 |
|------|------|
| `create_campaign` | 创建新战役 |
| `get_campaign` | 获取战役信息 |
| `list_campaigns` | 列出所有战役 |
| `delete_campaign` | 删除战役 |
| `get_campaign_summary` | 获取战役摘要（用于 LLM 上下文） |

### 5.2 角色管理

| Tool | 描述 |
|------|------|
| `create_character` | 创建角色/即兴 NPC |
| `get_character` | 获取角色信息 |
| `update_character` | 更新角色 |
| `list_characters` | 列出角色（可筛选 NPC） |
| `delete_character` | 删除角色 |

### 5.8 设计原则

1. **从 Client 角度设计**：MCP Tools 是 Client 需要执行的业务操作
2. **隐藏内部实现**：对话存储、上下文构建等由 Server 自动处理
3. **HP 变化是副作用**：`attack()`、`cast_spell()` 的结果包含 HP 变化
4. **时间推进是副作用**：`move_to()` 自动消耗时间

## 6. 与 Client 的职责边界

### 6.1 职责矩阵

| 职责 | Client | Server |
|------|:------:|:------:|
| 用户会话（战役）关联 | ✅ | ❌ |
| LLM 调用与编排 | ✅ | ❌ |
| WebSocket 推送 | ✅ | ❌ |
| 前端 API 网关 | ✅ | ❌ |
| 战役数据存储 | ❌ | ✅ |
| 角色数据管理 | ❌ | ✅ |
| 战斗状态管理 | ❌ | ✅ |
| 骰子/规则计算 | ❌ | ✅ |
| 地图数据管理 | ❌ | ✅ |
| 对话历史存储 | ❌ | ✅ |
| 上下文压缩 | ❌ | ✅ (简单策略) |

### 6.2 数据存储分离

```
Client 存储 (保持现有):
  ├─ client_sessions    会话元数据（名称、创建者、关联的 Server URL）
  └─ （可选缓存）       对话历史缓存

Server 存储 (PostgreSQL):
  ├─ campaigns          战役元数据
  ├─ characters         角色卡（属性、技能、HP）
  ├─ combats            战斗状态（参与者、回合、日志）
  ├─ maps               地图数据（大地图、战斗地图）
  ├─ messages           对话历史
  └─ game_states        游戏状态（时间、队伍位置）
```

### 6.3 Client 调整点

**需要调整的部分**（不在本次设计范围）：

1. **消息存储**：移除本地消息存储逻辑，改为调用 Server 的 `save_message`
2. **上下文获取**：新增调用 Server 的 `get_context` 或 `get_raw_context`
3. **游戏操作**：新增调用 Server 的游戏规则 Tools（attack、move 等）
4. **Session 模型**：`MCPServerURL` 字段已预留，需实际使用

### 7.2 地图层级关系

```
                    ┌─────────────────┐

    └─────┬──────┘    └─────┬──────┘    └─────┬──────┘
          │                 │                 │
          ▼                 ▼                 ▼
    ┌────────────┐    ┌────────────┐    ┌────────────┐
    │ 跃马旅店    │    │ 森林空地   │    │ 矿坑入口   │

### 7.3 设计原则

3. **图片前端管理**：Server 只存数据，图片由前端存储
4. **双地图统一格子系统**：大地图和战斗地图共享坐标系

## 8. 设计约束

### 8.1 技术约束

- Go 语言实现
- PostgreSQL 数据库
- MCP 协议 + HTTP 传输
- Server 不依赖 LLM
- 与 Client 架构模式一致（Handler → Service → Store）

### 8.2 业务约束

- 最终支持完整 5e 规则（循序渐进开发）
- 兼容其他 MCP Client
- 回合制战斗
- 战斗 AI 不在 Server 实现

### 8.3 设计约束

- 暂不考虑规则可选/冲突
- 暂不支持地图编辑
- 静态数据（法术、物品）依赖外部 RAG
- 状态效果系统后续扩展
