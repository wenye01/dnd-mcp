# 设计 Team

组织多角色协同工作，通过多角度讨论产出高质量设计包。**自动识别目标组件（Client/Server），采用相应设计策略**。处理用户反馈时需要人类决策。

## 目标组件识别

Team 会根据需求内容自动判断目标组件：

| 目标         | 识别关键词                              | 文档路径           | 设计重点                |
| ---------- | ---------------------------------- | -------------- | ------------------- |
| **Server** | 战斗、骰子、角色、怪物、规则、地图、回合、AC、HP、豁免、法术   | `docs/server/` | D&D 5e 规则、游戏引擎、状态管理 |
| **Client** | 会话、对话、消息、LLM、WebSocket、MCP、前端、用户界面 | `docs/client/` | 会话管理、对话历史、AI 编排     |
| **Both**   | 跨组件、端到端、整体架构                       | `docs/`        | 组件协作、接口契约           |

### 识别规则

```
1. 显式指定: "给 server 设计..." / "为 client 实现..."
2. 关键词匹配: 根据上表关键词判断
3. 默认规则:
   - 涉及 D&D 规则术语 → Server
   - 涉及 LLM/对话/MCP → Client
   - 模糊不清 → 询问用户
```

## 团队成员（7人）

| 角色         | 名称                    | 职责      | 侧重点                     |
| ---------- | --------------------- | ------- | ----------------------- |
| **需求分析师**  | Analyst               | 需求澄清与分析 | 用户诉求、场景分析、边界条件、**目标识别** |
| **系统架构师**  | Architect-System      | 整体架构设计  | 组件划分、技术选型、可扩展性、性能       |
| **领域架构师**  | Architect-Domain      | 领域模型设计  | 业务逻辑、领域模型、规则实现          |
| **集成架构师**  | Architect-Integration | 接口与集成设计 | API设计、数据流、外部系统集成        |
| **计划师**    | Planner               | 任务分解与规划 | 里程碑划分、任务定义、依赖关系         |
| **质量审查者**  | Reviewer-Quality      | 质量保证审查  | 可测试性、可维护性、错误处理、边界情况     |
| **一致性审查者** | Reviewer-Consistency  | 一致性保证审查 | 格式规范、文档完整、与现有设计的一致性     |

### 详细职责说明

#### Analyst（需求分析师）

- **核心任务**: 将模糊需求转化为清晰的需求规格，**识别目标组件**
- **输入**: 用户需求描述 / 反馈报告
- **输出**: 需求规格文档（/elicit 格式）+ **目标组件声明**
- **关注点**:
  - **目标组件识别**（Client/Server/Both）
  - 用户真正想要什么
  - 使用场景和边界条件
  - 非功能性需求
  - 与现有功能的关系

#### Architect-System（系统架构师）

- **核心任务**: 设计整体系统架构
- **输入**: 需求规格文档
- **输出**: 系统架构设计（/design 格式的架构部分）
- **关注点**:
  - 组件划分和职责
  - 技术选型和理由
  - 可扩展性和性能
  - 部署和运维考量
- **Client 侧重**: 会话管理、LLM 调用、WebSocket 连接
- **Server 侧重**: 规则引擎、状态管理、地图系统

#### Architect-Domain（领域架构师）

- **核心任务**: 设计领域模型和业务规则
- **输入**: 需求规格文档 + **D&D 5e 规则书（仅 Server）**
- **输出**: 领域模型设计（/design 格式的领域部分）
- **关注点**:
  - 领域模型和实体关系
  - 业务规则实现
  - 状态转换和生命周期
- **Client 侧重**: Session、Message、对话上下文
- **Server 侧重**: **D&D 5e 规则符合性**（必须查阅规则书）、Combat、Character、Dice

#### Architect-Integration（集成架构师）

- **核心任务**: 设计接口和集成方案
- **输入**: 需求规格文档 + 系统架构
- **输出**: 接口设计（/design 格式的接口部分）
- **关注点**:
  - API 设计（REST/MCP/内部接口）
  - 数据流和数据格式
  - 外部系统集成
  - 向后兼容性
- **Client 侧重**: HTTP API、WebSocket 协议、MCP 客户端接口
- **Server 侧重**: MCP 服务端接口、规则 API、游戏状态查询

#### Planner（计划师）

- **核心任务**: 将设计转化为可执行的开发计划
- **输入**: 整合后的设计文档 + **目标组件**
- **输出**: 开发计划（/plan 格式）
- **关注点**:
  - 里程碑划分
  - 任务分解（垂直切片）
  - 依赖关系
  - 风险评估
- **输出路径**（Client/Server 格式一致）:
  - 里程碑文件: `docs/{component}/plan/M{N}-{主题}.md`
  - README: `docs/{component}/plan/README.md`
  - 依赖图: `docs/{component}/plan/02-依赖关系.md`

#### Reviewer-Quality（质量审查者）

- **核心任务**: 从质量角度审查设计和计划
- **输入**: 设计文档 + 开发计划 + **目标组件**
- **输出**: 质量审查报告
- **审查清单**:
  - [ ] 可测试性：是否容易编写测试
  - [ ] 可维护性：代码结构是否清晰
  - [ ] 错误处理：异常情况是否覆盖
  - [ ] 边界情况：边界条件是否考虑
  - [ ] 性能考量：是否有性能风险
- **Client 额外检查**:
  - [ ] WebSocket 连接稳定性
  - [ ] LLM 调用超时处理
  - [ ] 会话并发安全
- **Server 额外检查**:
  - [ ] D&D 规则实现正确性
  - [ ] 游戏状态一致性
  - [ ] 规则边界条件覆盖

#### Reviewer-Consistency（一致性审查者）

- **核心任务**: 从一致性角度审查设计和计划
- **输入**: 设计文档 + 开发计划 + 现有文档 + **目标组件**
- **输出**: 一致性审查报告
- **审查清单**:
  - [ ] 格式规范：是否符合 /design、/refine、/plan 格式
  - [ ] 文档完整：是否缺少必要章节
  - [ ] 命名一致：与现有代码命名风格一致
  - [ ] 架构一致：不与现有架构冲突
  - [ ] 术语一致：使用项目统一术语
- **Client 参考文档**: `docs/client/详细设计.md`, `docs/client/重构计划.md`
- **Server 参考文档**: `docs/server/详细设计.md`, `docs/server/关键技术点讨论.md`
- **规则参考**: `docs/dnd5e规则书/查询指南.md`（仅 Server）

## 核心原则

1. **人类决策优先** - 所有非 Bug 类反馈都需要人类决策
2. **新增而非修改** - 增量计划独立于现有计划，不修改已有文档
3. **格式规范** - 输出严格遵循现有 skill 格式要求
4. **多角度讨论** - 架构师从不同视角设计，审查者从不同维度检查

## 模式

| 模式          | 输入   | 输出    | 说明          |
| ----------- | ---- | ----- | ----------- |
| `init`      | 需求描述 | 完整设计包 | 初始设计流程      |
| `feedback`  | 反馈报告 | 决策点文档 | 反馈分析，等待人类决策 |
| `implement` | 决策确认 | 增量计划  | 根据人类决策生成计划  |
| `adjust`    | 需求描述 | 设计调整方案 | 根据新需求调整/扩展现有设计 |

## 使用示例

```
# Server 设计（自动识别：包含"战斗"关键词）
/design-team init 实现一个战斗回合管理系统

# Client 设计（自动识别：包含"会话"关键词）
/design-team init 为 client 实现会话持久化功能

# 显式指定目标组件
/design-team init --target server 实现角色属性计算
/design-team init --target client 优化 LLM 调用链路

# 处理反馈（会停止等待人类决策）
/design-team feedback docs/体验报告-20240101.md

# 人类决策后，生成增量计划
/design-team implement 决策点-骰子历史记录.md --decision A

# 根据新需求调整/扩展现有设计
/design-team adjust 为战斗系统添加借机攻击规则
/design-team adjust --target client 支持多轮对话上下文压缩
/design-team adjust 扩展骰子系统支持命运骰(Fate Dice)
```

## 设计调整流程 (adjust 模式)

**适用场景**: 用户有明确的新需求，需要调整或扩展现有设计。

```
┌─────────────────────────────────────────────────────────────┐
│ Step 1: 接收需求描述                                         │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ Step 2: 需求分析 (Analyst)                                   │
│                                                              │
│ Analyst:                                                     │
│ - 识别目标组件 (Client/Server/Both)                          │
│ - 分析需求类型:                                              │
│   ├── 新增功能 → 扩展设计                                    │
│   ├── 修改行为 → 调整设计                                    │
│   └── 补充细节 → 完善设计                                    │
│ - 评估与现有功能的关系                                       │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ Step 3: 现有设计审查                                          │
│                                                              │
│ - 读取现有设计文档 (docs/{component}/详细设计.md)            │
│ - 读取现有开发计划 (docs/{component}/plan/)                  │
│ - 识别受影响的部分                                           │
│ - Server: 检查 D&D 5e 规则书相关内容                         │
└─────────────────────────────────────────────────────────────┘
                            │
```

## 反馈分类标准

### Bug (直接转开发 Team)

```
判断标准：
- 功能行为与设计不符
- 系统报错或崩溃
- 数据处理错误

处理方式：
1. 确认是 Bug
2. 标记优先级 (P0/P1/P2)
3. 直接转交开发 Team
4. 不进入设计流程

示例：
✓ "点击按钮后 500 错误" → Bug
✓ "返回的数据缺少字段" → Bug
✗ "按钮位置不好找" → 体验问题 (非 Bug)
```

### 体验问题 (需要人类决策)

```
判断标准：
- 功能正确但体验不佳
- 用户流程不顺畅
- 可用性需要改进

处理方式：
1. Analyst 分析问题
2. Team 讨论改进方案
3. 生成决策点文档
4. 等待人类决策
5. 根据决策生成增量计划

示例：
✓ "错误提示太技术化，用户看不懂" → 体验问题
✓ "完成一个任务需要点击太多次" → 体验问题
```

### 新需求想法 (需要人类决策)

```
判断标准：
- 用户期望的新功能
- 现有功能的扩展
- 新的使用场景

处理方式：
1. Analyst 澄清需求
2. Team 讨论可行性和方案
3. 生成决策点文档
4. 等待人类决策
5. 根据决策生成增量计划

示例：
✓ "希望能导出历史记录" → 新需求
✓ "希望能批量导入数据" → 新需求
```

## 执行流程

### /design-team init [需求]

```
1. 解析需求描述
2. Spawn Analyst:
   a. 识别目标组件 (Client/Server/Both)
   b. 生成需求规格 (/elicit 格式)
   c. 声明目标组件和输出路径

3. 根据目标组件加载参考资料:
   - Server: 加载 D&D 5e 规则书查询指南
   - Client: 加载现有 client 设计文档
   - Both: 两者都加载

4. 并行 Spawn 3 名架构师:
   - Architect-System: 系统架构视角
   - Architect-Domain: 领域模型视角 (Server 侧重复核 D&D 规则)
   - Architect-Integration: 集成接口视角

5. 架构师团队讨论，整合方案

6. Spawn Planner:
   - 根据目标组件选择输出路径
   - 生成开发计划 (/plan 格式)

7. 并行 Spawn 2 名审查者:
   - Reviewer-Quality: 质量视角审查
   - Reviewer-Consistency: 一致性视角审查

8. 审查者反馈，修正设计

9. 综合输出到正确路径:
   - Server → docs/server/
   - Client → docs/client/

10. 请求用户审核
11. 用户批准后，通知开发 Team
```

### /design-team feedback [反馈报告]

```
1. 读取反馈报告
2. 分类反馈:
   - Bug → 直接转开发 Team，结束
   - 体验问题/新需求 → 继续

3. Spawn Analyst 分析反馈:
   a. 识别反馈涉及的目标组件
   b. 分析影响范围
   c. 确定输出路径

4. 根据目标组件加载参考资料:
   - Server: 加载 D&D 5e 规则书查询指南
   - Client: 加载现有 client 设计文档

5. 并行 Spawn 3 名架构师分析影响:
   - Architect-System: 架构影响
   - Architect-Domain: 领域规则影响 (Server 侧重 D&D 规则)
   - Architect-Integration: 接口集成影响

6. 组织内部讨论 (Analyst + 3 Architects + Planner)

7. 并行 Spawn 2 名审查者审查方案:
   - Reviewer-Quality: 方案质量
   - Reviewer-Consistency: 与现有设计一致性

8. 生成决策点文档 (包含目标组件声明)

9. 【停止】提示用户需要决策

10. 等待用户决策
```

### /design-team implement [决策点文档] --decision [A/B/C]

```
1. 读取决策点文档
2. 确认人类决策
3. 如果选择 "不做":
   - 关闭决策点
   - 记录决策原因
   - 结束
4. 如果选择方案:
   - 并行 Spawn 3 名架构师生成增量设计（各自视角）
   - 整合增量设计
   - Spawn Planner 生成增量计划
   - 并行 Spawn 2 名审查者审查
   - 检查与现有计划的集成
   - 通知开发 Team 新增任务
```
