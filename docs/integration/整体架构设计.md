## 2. 组件职责划分

### 2.1 职责矩阵

| 职责 | Client | Server | 说明 |
|------|:------:|:------:|------|
| 用户会话管理 | ✅ | ❌ | Session/Campaign 映射 |
| 对话历史管理 | ❌ | ✅ | Server 统一存储 |
| LLM 调用 | ✅ | ❌ | GLM-4.7-Flash |
| 工具调用编排 | ✅ | ❌ | 处理 Tool Call |
| 规则执行 | ❌ | ✅ | D&D 5e 规则 |
| 角色数据管理 | ❌ | ✅ | Character CRUD |
| 战斗状态管理 | ❌ | ✅ | Combat State |
| 骰子投掷 | ❌ | ✅ | Dice Roll |
| WebSocket 推送 | ✅ | ❌ | 实时事件 |

### 2.2 Client 职责详解

```
Client 核心职责:
├── 会话管理
│   ├── 创建/查询/删除 Session
│   └── Session 与 Campaign 映射
├── LLM 编排
│   ├── 构建 System Prompt
│   ├── 调用 GLM-4.7-Flash
│   ├── 处理流式响应
│   └── 处理 Tool Call
├── Server 协调
│   ├── MCP 协议连接
│   ├── 工具调用转发
│   └── 结果整合
└── 实时推送
    ├── WebSocket 连接管理
    └── 事件广播
```

### 2.3 Server 职责详解

```
Server 核心职责:
├── 战役管理 (M2)
│   ├── create_campaign
│   ├── get_campaign
│   └── list_campaigns
├── 角色管理 (M3)
│   ├── create_character
│   ├── get_character
│   └── modify_hp
├── 骰子系统 (M4)
│   ├── roll_dice
│   ├── roll_check
│   └── roll_attack
├── 战斗系统 (M5)
│   ├── start_combat
│   ├── next_turn
│   └── end_combat
└── 数据存储
    ├── Redis 缓存
    └── PostgreSQL 持久化
```

## 3. 技术选型

### 3.1 GLM-4.7-Flash 集成方案

```go
// LLM 客户端配置
type GLMConfig struct {
    APIEndpoint string  // https://open.bigmodel.cn/api/paas/v4/chat/completions
    Model       string  // glm-4.7-flash
    APIKey      string  // 从环境变量读取
    MaxTokens   int     // 65536
    Temperature float64 // 1.0
}

// 工具调用格式（OpenAI 兼容）
type Tool struct {
    Type     string                 `json:"type"`     // "function"
    Function ToolFunction           `json:"function"`
}

type ToolFunction struct {
    Name        string                 `json:"name"`
    Description string                 `json:"description"`
    Parameters  map[string]interface{} `json:"parameters"`
}
```

### 3.2 MCP 协议实现方案

```
MCP 协议选项:
├── stdio (推荐用于单机部署)
│   ├── Client 启动 Server 子进程
│   └── 通过 stdin/stdout 通信
└── HTTP (可选，用于远程部署)
    ├── Server 作为 HTTP 服务运行
    └── Client 通过 HTTP 调用

本设计采用: stdio 方式
理由:
1. 简单可靠，适合单机部署
2. 无需额外端口
3. 与 MCP 标准实现一致
```

## 4. 部署架构

### 4.1 单机部署方案

```
┌─────────────────────────────────────────────────────────────────┐
│                          主机 (Windows/Linux)                   │
│                                                                 │
│  Redis :6379      PostgreSQL :5432                             │
│                                                                 │
│  MCP Server (子进程)          MCP Client (:8080)               │
│       ↑                              │                          │
│       │ stdio                       │ HTTP/WS                   │
│       └──────────────────────────────┘                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 环境配置

```bash
# 数据库
REDIS_HOST=localhost:6379
DATABASE_URL=postgres://user:pass@localhost:5432/dnd_mcp

# GLM-4.7-Flash
GLM_API_KEY=your-api-key
GLM_API_ENDPOINT=https://open.bigmodel.cn/api/paas/v4/chat/completions
GLM_MODEL=glm-4.7-flash

# Client
HTTP_HOST=0.0.0.0
HTTP_PORT=8080

# Server (通过 MCP 配置)
MCP_SERVER_COMMAND=./bin/dnd-server
```

### 4.3 启动顺序

```
1. Redis / PostgreSQL
   └── 确保数据存储可用

2. MCP Server (独立启动或由 Client 启动)
   └── 启动 Server 进程

3. MCP Client
   └── 连接存储 → 连接 Server → 启动 HTTP 服务

4. 前端 (可选)
   └── 连接 Client API
```

## 5. 数据流设计

### 5.1 用户消息处理流程

```
┌───────────────────────────────────────────────────────────────────────────┐
│                        用户消息处理流程                                    │
└───────────────────────────────────────────────────────────────────────────┘

用户输入: "我攻击哥布林"
    │
    ▼
┌─────────────────┐
│ 1. 接收消息     │ Client HTTP API
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 2. 构建上下文   │ 获取对话历史 + 游戏状态
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 3. 调用 LLM     │ GLM-4.7-Flash API
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 4. 解析响应     │ 检查是否有 Tool Call
└────────┬────────┘
         │
    ┌────┴────┐
    │         │
    ▼         ▼
[有 Tool]  [无 Tool]
    │         │
    ▼         │
┌───────────┐ │
│ 5. 调用   │ │
│    Server │ │
└─────┬─────┘ │
      │       │
      ▼       │
┌───────────┐ │
│ 6. 获取   │ │
│    结果   │ │
└─────┬─────┘ │
      │       │
      └───┬───┘
          │
          ▼
┌─────────────────┐
│ 7. 生成响应     │ 整合结果，生成自然语言
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 8. 保存消息     │ 保存对话历史
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 9. 返回用户     │ HTTP 响应 / WebSocket 推送
└─────────────────┘
```

### 5.2 工具调用流程

```
LLM 返回 Tool Call:
{
  "tool_calls": [{
    "id": "call_123",
    "function": {
      "name": "roll_attack",
      "arguments": "{\"target\":\"goblin_1\",\"weapon\":\"longbow\"}"
    }
  }]
}

    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│ Client 处理:                                                 │
│ 1. 解析 tool_calls                                          │
│ 2. 映射到 Server MCP Tool                                   │
│ 3. 转换参数格式                                             │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
MCP Request:
{
  "method": "tools/call",
  "params": {
    "name": "roll_attack",
    "arguments": {
      "target": "goblin_1",
      "weapon": "longbow"
    }
  }
}
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│ Server 处理:                                                 │
│ 1. 执行攻击检定                                             │
│ 2. 计算伤害                                                 │
│ 3. 更新目标 HP                                              │
│ 4. 返回结果                                                 │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
MCP Response:
{
  "content": [{
    "type": "text",
    "text": "{\"hit\":true,\"roll\":18,\"damage\":8,\"target_hp\":12}"
  }]
}
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│ Client 处理:                                                 │
│ 1. 解析结果                                                 │
│ 2. 构建工具结果消息                                         │
│ 3. 再次调用 LLM 生成自然语言响应                            │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
最终响应: "你射出一箭，命中哥布林！造成 8 点伤害。哥布林还有 12 点生命值。"
```

## 7. LLM System Prompt 设计

### 7.1 AI DM System Prompt

```markdown
你是一个经验丰富的《龙与地下城》第五版（D&D 5e）地下城主（DM）。
你的职责是通过自然语言与玩家互动，引导他们进行冒险。

## 你的能力

你可以使用以下工具来执行游戏规则：

### 战役管理
- create_campaign: 创建新的战役/冒险
- get_campaign: 获取当前战役信息

### 角色管理
- create_character: 创建玩家角色或 NPC
- get_character: 获取角色详细信息
- modify_hp: 修改角色生命值

### 骰子与检定
- roll_dice: 投掷任意骰子
- roll_check: 执行属性或技能检定
- roll_attack: 执行攻击检定
- roll_damage: 执行伤害检定

### 战斗管理
- start_combat: 初始化战斗
- get_combat_state: 获取当前战斗状态
- next_turn: 推进到下一回合
- end_combat: 结束战斗

## 行为准则

1. **始终使用工具执行规则**: 当玩家声明动作时，调用相应的工具
2. **描述生动**: 用生动的语言描述场景和结果
3. **公平公正**: 严格按照 D&D 5e 规则执行
4. **鼓励探索**: 引导玩家探索世界

## 响应格式

当需要执行游戏规则时：
1. 调用相应的工具
2. 等待工具返回结果
3. 根据结果生成自然语言描述

不要直接编造骰子结果或游戏数值，必须通过工具获取。
```
