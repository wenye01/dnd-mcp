## 二、关键发现

### 🔴 P0 阻塞问题：API 协议不匹配

**问题描述**：
- **Client** 的 `server.HTTPClient` 实现使用 REST API 风格端点：
  - `GET /api/campaigns/{id}/context`
  - `POST /api/campaigns/{id}/messages`
  - `POST /api/campaigns/{id}/tools/{toolName}`

- **Server** 只提供 MCP 协议 API：
  - `POST /mcp/initialize`
  - `GET /mcp/tools`
  - `POST /mcp/tools/call`

**影响**：
- Client 无法通过 `ServerClient` 保存消息到 Server
- 错误信息：`保存消息失败 (status 404): 404 page not found`

**验证**：
```bash
# Client 尝试保存消息（失败）
POST http://localhost:8080/api/campaigns/{id}/chat
-> 404 page not found

# Server MCP API（正常工作）
POST http://localhost:8080/mcp/tools/call
{"name": "create_campaign", "arguments": {...}}
-> 成功返回 campaign 数据
```

---

## 三、根本原因分析

### 设计文档 vs 实现

| 方面 | 设计文档 (M7.5) | 实际实现 |
|------|----------------|----------|
| ServerClient 接口 | 调用 Server HTTP API | HTTPClient 使用 REST 风格 |
| Server API | MCP 协议 | 仅 MCP 协议（无 REST API） |
| 通信方式 | 未明确指定 | REST vs MCP 不匹配 |

**问题根源**：
1. M7.5 设计文档假设 Server 会提供 REST API
2. Server 实际上只实现了 MCP 协议 API
3. Client 的 `HTTPClient` 按 REST API 设计，与 Server 不兼容

---

## 四、解决方案建议

### 方案 A：Client 改用 MCP 协议（推荐）

**修改范围**：仅 Client

**实现方式**：
1. 修改 `server.HTTPClient`，使用 MCP 协议调用 Server
2. 端点映射：
   - `GetContext` -> `POST /mcp/tools/call` with `name: "get_context"`
   - `SaveMessage` -> `POST /mcp/tools/call` with `name: "save_message"`
   - `CallTool` -> `POST /mcp/tools/call`

**优点**：
- 修改范围小
- 复用 Server 现有实现
- 符合 MCP 协议设计理念

### 方案 B：Server 添加 REST API

**修改范围**：Server

**实现方式**：
1. 在 Server 添加 REST API 端点
2. 实现 `/api/campaigns/{id}/context` 等端点

**缺点**：
- 修改范围大
- 需要维护两套 API
